# MXFP8 C8 KV 缓存量化设计文档

## 1. 背景与目标

### 1.1 背景

在大语言模型推理中，KV Cache 是显存占用的主要来源之一。随着序列长度增长，KV Cache 的显存开销呈线性增长，成为限制推理吞吐量的关键瓶颈。

华为 Ascend NPU（如 910B/C、A5）支持 FP8（E4M3）数据格式和 MXFP8（Microscaling FP8）量化能力。MXFP8 采用分组量化策略，每 32 个元素共享一个 E8M0 缩放因子，在保持精度的同时将 KV Cache 的存储开销压缩至原来的 50%。

### 1.2 目标

- 实现 **MXFP8 C8 KV Cache 量化**，将 K/V 从 BF16/FP16 量化为 FP8 E4M3 格式存储
- 利用 NPU 的 `npu_fused_infer_attention_score_v2` 算子，在注意力计算中直接消费 FP8 KV Cache + E8M0 缩放因子，避免反量化开销
- 支持 Prefill、Decode、ChunkedPrefill 三种推理阶段
- 处理 V 轴 axis=0 量化时的尾部窗口（Tail Window）问题

**两种 C8 量化路径的判断逻辑**（`modelslim_config.py:510-514`）：

```python
if self.quant_description.get("kv_cache_type") == "C8":
    # INT8 静态 per-channel 量化
    return AscendC8KVCacheAttentionMethod(...)
elif is_c8_mxfp_kv_quant(vllm_config):
    # MXFP8 动态 per-group 量化

```

### 3.2 量化方法层

#### AscendC8KVCacheAttentionMethod（INT8 静态量化）

- **文件**：`vllm_ascend/quantization/methods/kv_c8.py:99-155`
- **量化方式**：per-channel INT8，使用静态 scale/offset
- **KV Cache dtype**：`torch.int8`
- **权重**：`k_cache_scale`, `k_cache_offset`, `v_cache_scale`, `v_cache_offset`
- **类替换**：将 `layer.impl` 替换为 `AscendC8AttentionBackendImpl`

#### AscendC8MXFPKVCacheAttentionMethod（MXFP8 动态量化）

- **文件**：`vllm_ascend/quantization/methods/kv_c8.py:158-194`
- **量化方式**：per-group FP8 E4M3 + E8M0 scale，每 64 个元素一组
- **KV Cache dtype**：`torch.float8_e4m3fn`
- **类替换**：将 `layer.impl` 替换为 `AscendC8MXFPAttentionBackendImpl`
- **无需额外权重**：scale 在推理时动态计算

### 3.4 尾部窗口管理

- **文件**：`vllm_ascend/device/mxfp_tail_window.py`
- **类**：`MxfpTailWindowWriter`
- **问题背景**：V 的 MXFP8 量化沿 axis=0（token 维度）进行，每 64 个 token 一组。Decode 阶段每次只生成 1 个 token，不足 64 个时需要保留未满组的原始 BF16 V 值，等凑满 64 后再量化写入 cache。

**数据结构**：

```
win_v:    [max_num_seqs, 64, num_kv_heads, v_dim]   # 尾部窗口的 BF16 V 值
win_slots: [max_num_seqs, 64]                        # 对应的 slot mapping
win_lens:  [max_num_seqs]                            # 每个请求的窗口长度 (0~63)
```

**生命周期**：

```
Prefill → save_prefill_tail() → 保存末尾不满 64 的部分
Decode  → 拼接窗口 + 新 token → 量化 → 重写 cache → 更新窗口
请求结束 → prune() → 清理窗口
```

### 3.5 KV Cache 布局与 Scatter 操作

- **文件**：`vllm_ascend/device/mxfp_compat.py`

**KV Cache 四张表的布局**：

| 张量 | 布局 | 说明 |
|------|------|------|
| `k_cache` | `[num_blocks, block_size, num_kv_heads, head_dim]` | FP8 E4M3 K 值 |
| `v_cache` | `[num_blocks, block_size, num_kv_heads, v_dim]` | FP8 E4M3 V 值 |
| `k_scale_cache` | `[num_blocks, num_kv_heads, block_size, head_dim//64, 2]` | K 的 E8M0 缩放因子 |
| `v_scale_cache` | `[num_blocks, num_kv_heads, block_size//64, head_dim, 2]` | V 的 E8M0 缩放因子 |

**Scatter 操作**：

| 函数 | 功能 |
|------|------|
| `scatter_mxfp_k_scale_cache()` | 按 slot_mapping 将 K scale 写入 paged cache |
| `scatter_mxfp_v_cache()` | 使用 `npu_scatter_nd_update_` 将 FP8 V 写入 cache |
| `scatter_mxfp_v_scale_cache()` | 按 64-token 组映射将 V scale 写入 paged cache |

### 3.6 KV Cache 内存规划

- **文件**：`vllm_ascend/patch/platform/patch_kv_cache_interface.py`
- **功能**：Patch 上游 `FullAttentionSpec.real_page_size_bytes`，使 KV Cache 内存预算包含 FP8 K/V + E8M0 scale 的总开销

**每页内存计算**（`mxfp_kv_page_size_bytes`）：

```
kv_bytes   = block_size × num_kv_heads × (k_dim + v_dim) × 1  (FP8 = 1 byte)
scale_bytes = (k_scale_page_bytes + v_scale_page_bytes) × 1    (E8M0 = 1 byte)
total      = kv_bytes + scale_bytes
```

## 4. 量化/反量化流程

### 4.1 MXFP8 量化流程

```
BF16 Tensor → npu_dynamic_mx_quant(dst_type=float8_e4m3fn, axis=0)
            → (FP8 E4M3 Tensor, E8M0 Scale Tensor)
```

- **Q 量化**：`axis` 默认，per-32-element 分组
- **K 量化**：`axis` 默认，per-32-element 分组
- **V 量化**：`axis=0`，沿 token 维度 per-64-element 分组（与 FIA V2 的 value_quant_mode=8 对应）

### 4.2 Prefill 阶段 V 量化

```
1. 计算每个请求的 padded_len = ceil(num_tokens / 64) × 64
2. 将所有请求的 V 拼接到一个 padded_value 张量（零填充对齐）
3. 单次调用 npu_dynamic_mx_quant(padded_value, axis=0)
4. 从批量结果中按偏移量切分出每个请求的 FP8 V 和 scale
5. 调用 save_prefill_tail() 保存末尾不满 64 的部分
```

**零填充安全性**：MXFP8 的 E8M0 scale 取组内绝对最大值，零不会增大最大值，因此零填充不影响有效 token 的量化精度。

### 4.3 Decode 阶段 V 量化（尾部窗口）

```
1. 读取每个 decode 请求的尾部窗口 (win_v, win_slots, win_len)
2. 拼接 [窗口 V | 新 V]，pad 到 64 的倍数
3. 调用 npu_dynamic_mx_quant(padded_value, axis=0)
4. 将重量化后的窗口部分写回 v_cache（_rewrite_v_window_to_cache）
5. 更新尾部窗口状态：
   - win_len + new_tokens == 64 → 窗口清空
   - win_len + new_tokens < 64  → 追加到窗口
   - win_len + new_tokens > 64  → 保留余数
```

### 4.4 FIA V2 反量化（算子内部）

`npu_fused_infer_attention_score_v2` 在 NPU 内部完成：

```
FP8 Q × E8M0 Q_scale → BF16 Q
FP8 K × E8M0 K_scale → BF16 K
FP8 V × E8M0 V_scale → BF16 V
Attention(Q, K, V) → BF16 output
```

## 5. NPU 算子映射

| 算子 | 用途 | 调用位置 |
|------|------|---------|
| `npu_dynamic_mx_quant` | BF16 → FP8 E4M3 + E8M0 scale 动态量化 | forward(), _mxfp_quant_value() |
| `npu_fused_infer_attention_score_v2` | FP8 KV Cache 注意力计算 | _run_mxfp8_fia_v2() |
| `npu_scatter_nd_update_` | FP8 V 值 scatter 写入 cache | scatter_mxfp_v_cache() |
| `DeviceOperator.reshape_and_cache` | FP8 K/V 值 reshape 并写入 paged cache | reshape_and_cache() |

**FIA V2 关键参数**：

```python
npu_fused_infer_attention_score_v2(
    query, key, value,                    # FP8 E4M3
    dequant_scale_query/key/value,        # E8M0 scale
    query_quant_mode=6,                   # MXFP8 per-group
    key_quant_mode=6,                     # MXFP8 per-group
    value_quant_mode=8,                   # MXFP8 axis=0 per-64-group
    query/key/value_dtype=float8_e4m3fn,
    dequant_scale_*_dtype=float8_e8m0fnu,
    input_layout="TND",
    block_table=...,
    sparse_mode=3 (causal) / 0 (decode),
)
```

## 6. 与现有 Attention 后端的对比

| 特性 | AscendAttentionBackendImpl | AscendC8AttentionBackendImpl | AscendC8MXFPAttentionBackendImpl |
|------|---------------------------|------------------------------|----------------------------------|
| KV Cache dtype | BF16/FP16 | INT8 | FP8 E4M3 |
| 量化方式 | 无 | 静态 per-channel | 动态 per-group (MXFP8) |
| Scale 存储 | 无 | 4 个参数 (k/v scale/offset) | 2 张额外 cache (k/v scale cache) |
| FIA 算子 | `npu_fused_infer_attention_score` V1 | V1 + antiquant 参数 | `npu_fused_infer_attention_score_v2` |
| KV Cache 转置 | 否 | 否 | 是（BNSD 布局） |
| 尾部窗口 | 不需要 | 不需要 | 需要（V axis=0 量化） |
| block_size 要求 | 无特殊要求 | 无特殊要求 | A5 要求 512 |
| ACL Graph | 支持 | 支持 | 不支持 |
| Non-causal | 支持 | 支持 | 不支持 |
| Sliding Window | 支持 | 支持 | 不支持 |

## 7. 关键数据结构

### 7.1 MXFP KV Scale 布局常量

```python
MXFP_KV_SCALE_GROUP_SIZE = 64    # V axis=0 量化的分组大小
MXFP_KV_SCALE_VALUES_PER_GROUP = 2  # 每组的 E8M0 scale 元素数
MXFP8_GROUP_SIZE = 32             # Q/K per-element 量化的分组大小
MXFP_SCALE_DTYPE_SIZE = 1        # E8M0 scale 每元素 1 字节
```

### 7.2 KV Cache 张量元组

MXFP8 模式下 `kv_cache` 为 4 元素元组：

```python
kv_cache = (
    k_cache,        # [num_blocks, block_size, num_kv_heads, head_dim]  FP8 E4M3
    v_cache,        # [num_blocks, block_size, num_kv_heads, v_dim]     FP8 E4M3
    k_scale_cache,  # [num_blocks, num_kv_heads, block_size, head_dim//64, 2]  uint8 (E8M0)
    v_scale_cache,  # [num_blocks, num_kv_heads, block_size//64, head_dim, 2]   uint8 (E8M0)
)
```

## 8. 已知限制

1. **不支持 Non-causal Attention**：C8_MXFP 目前仅支持因果注意力，双向注意力（如 Embedding 模型）不可用
2. **不支持 Sliding Window**：滑动窗口注意力尚未实现
3. **不支持 ACL Graph**：无法使用图捕获加速，因为 V 量化的尾部窗口逻辑是动态的
4. **不支持 Hamming Sparse KV 压缩**：与 KV 压缩特性不兼容
5. **block_size 约束**：A5 平台强制要求 `block_size=512`，且 `head_dim` 和 `block_size` 必须被 64 整除
6. **V scale slot mapping 一致性要求**：`scatter_mxfp_v_scale_cache` 要求同一 64-token 组内的所有 slot 属于同一 block，否则会报错
7. **Chunked Prefill 的 V 量化**：当前 prefill 路径已实现批量 pad 量化，但 chunked prefill 的完整优化尚未完成（代码中有 TODO 标注）
